Motivation

The Problem

What are we solving?

  • Predict the nightly price of an NYC Airbnb listing from its features
  • Target: log(price), reduces right-skew from 19.1 to 0.6
  • Supervised regression evaluated by R², RMSE (log), and MAE ($)
  • 80/20 train/test split, random seed 42 for reproducibility

Why does it matter?

  • Hosts need pricing guidance: underpricing loses revenue, overpricing loses bookings
  • Travelers want to assess whether a listing is fairly priced
  • NYC has 48,000+ active listings; even a $5 improvement in pricing accuracy has large aggregate impact

Broader use cases

  • Dynamic pricing platforms: automate host price suggestions
  • Real estate valuation: short-term rental income estimation
  • Market analysis: track neighbourhood-level price trends over time
Core question: which features actually drive Airbnb prices, and does more model complexity help?

Prior Work

How Has This Been Solved Before?

Existing approaches

  • Most Kaggle solutions use plain OLS or a single RF with basic numeric features only (R² ≈ 0.50–0.58)
  • Some add borough one-hot encoding; very few address the 221-neighbourhood high-cardinality problem
  • Text features (listing name/description) are almost always discarded entirely
  • No complexity ladder: models are compared in isolation, making it hard to attribute performance gains

How our approach differs

  • Smoothed target encoding collapses 221 neighbourhoods into 1 meaningful feature
  • TF-IDF + PCA on listing names extracts 128 latent text signals (novel)
  • 12 keyword flags capture semantic listing descriptors (luxury, studio, cozy…)
  • Complexity ladder across 5 models isolates the contribution of each design choice

What we gain

  • Our baseline linear model (R² = 0.595) already exceeds typical published baselines (0.50–0.58)
  • XGBoost (R² = 0.656) pushes further via sequential residual correction
  • Feature importance analysis reveals why models predict what they do, not just how well
  • End-to-end reproducible pipeline: fixed seeds, modular notebooks, exported CSVs
Prior work leaves signal on the table by ignoring text and neighbourhood structure. Our pipeline directly addresses both gaps.

Structured features (7)

  • Latitude, longitude (geographic signal)
  • Room type → 2 one-hot dummies
  • Borough → 4 one-hot dummies
  • Availability, minimum nights, review counts
  • Host listing count

Neighbourhood encoding (1)

  • 221 unique levels → 1 column via smoothed target encoding
  • Smoothing λ = 20 shrinks rare levels toward global mean
  • Encodes: mean log_price per neighbourhood on training data only

Text features (142)

  • 12 binary keyword flags (luxury, cozy, studio, loft…)
  • Bedroom count extracted via regex, imputed at median
  • TF-IDF (200 terms, bigrams) + PCA → 128 components (80% variance)
  • 156 total features entering the models

Novelty

High-Cardinality Encoding

221 unique neighbourhoods: how do we encode them without losing signal or exploding dimensionality?

One-hot (220 cols)

  • Rare neighbourhoods (Fort Wadsworth, n=1) get unstable coefficients
  • Massive sparse feature space
  • Multicollinearity risk

Target Encoding ✓ Our choice

  • Single column: smoothed mean of log_price per neighbourhood
  • Rare levels pulled toward global mean
  • Tribeca: 5.59  ·  Corona: 4.12

Drop column

  • Throws away real price signal
  • Williamsburg ≠ Bushwick in pricing
  • Not a real option
enc(c) =   ( n_c · ȳ_c + 20 · ȳ ) / ( n_c + 20 )   ·   smoothing λ = 20

Models

Five Models: Increasing Complexity

Baseline
Linear Regression (OLS)  ·  reference point, no regularisation · features standardised (StandardScaler)
Regularised
Ridge (L2)  ·  shrinks all coefficients smoothly · α = 48.33 selected via 3-fold CV on training set
Lasso (L1)  ·  zeros out irrelevant features · α = 0.00044 via 3-fold CV · 148 / 156 features retained
Non-linear
Random Forest  ·  200 trees · min_samples_leaf = 5 · max_features = sqrt(p) · unscaled features
Boosting
XGBoost  ·  lr = 0.05 · max_depth = 5 · subsample = 0.8 · n_trees = 328 via early stopping on 15% hold-out
Why not SVM or neural networks? SVMs scale poorly to n = 39k with 156 features (O(n²) kernel). Neural networks require more data and careful tuning for marginal gain on structured tabular problems. These five models efficiently span the bias-variance trade-off and remain interpretable.

R² interpretation

  • Proportion of price variance explained by features
  • R² = 0.656 means XGBoost explains 65.6% of price variation
  • Remaining 34% driven by unmeasured factors

MAE interpretation

  • Mean absolute error in dollar space after exp-transforming predictions
  • XGBoost: typical prediction off by $52.91 on a $152 mean listing
  • Prefer MAE over RMSE: robust to the handful of $1,000+ luxury outliers

Key observation

  • Linear models all converge within 0.01% R² of each other
  • XGBoost gains +6% R² over linear — the only meaningful jump
  • Performance ceiling is the data, not the algorithm choice

Findings

What Drives Airbnb Prices?

Linear model (Ridge coefficients)

  • Room: Private: strongest negative effect, coef = −0.32 (private rooms cost ~27% less than entire homes)
  • Neighbourhood: target-encoded signal, +0.22
  • Borough: Manhattan: +0.11 vs. Bronx baseline
  • Bedroom Count: +0.08 per extra bedroom
  • Keyword: Studio: −0.07 (negatively priced signal)

Tree models (feature importance)

  • Room: Private: 16–38% of model decisions across RF and XGBoost
  • Neighbourhood: 5–8% (Tribeca vs. Corona = 1.5x price difference)
  • Longitude: 5.9% in RF — captures Manhattan's east-west price gradient
  • Top 5 features account for ~35% of RF decisions combined

Text feature contribution

  • TF-IDF PCs and keyword flags appear in both RF and XGBoost top-15 lists
  • kw_studio: consistent negative price signal across all models
  • kw_luxury: small positive signal (+0.04 in Ridge)
  • Text adds marginal but real signal: removing it drops R² by ~0.5%
Two completely different algorithm families agree: room type and neighbourhood dominate NYC pricing.

Discussion

Why Did the Models Converge?

Why regularisation did not help

  • n = 39k >> p = 156, so OLS is already well-conditioned
  • PCA on TF-IDF removed collinearity before modelling
  • Lasso barely penalised (α ≈ 0.00044, kept 148/156 features)

Why RF scored slightly lower

  • Log-transform already linearised the main price signal
  • 128 sparse TF-IDF PCs favour linear models over trees

Comparison to published work

  • Typical Kaggle baselines on this dataset: R² ≈ 0.50–0.58 (numeric features only)
  • Our linear model (R² = 0.595) already beats most published baselines
  • XGBoost (R² = 0.656) is competitive with best reported results on this dataset

Limitations

  • ~40% of price variance is unmeasured (photos, amenities, host quality)
  • Spatial autocorrelation present but not explicitly modelled
  • Luxury listings ($1,000+) systematically under-predicted

Residuals: Ridge, RF, XGBoost

Conclusion

Summary

What we did

  • End-to-end ML pipeline on 48,870 NYC Airbnb listings
  • Engineered 142 text features from listing names (TF-IDF + PCA + keyword flags)
  • Compared 5 models across a complexity ladder
  • Smoothed target encoding for 221 neighbourhood levels
  • Best result: XGBoost R² = 0.656, MAE = $52.91

Key takeaways

  • Room type is the single most powerful predictor across all models
  • More model complexity does not guarantee better performance
  • The data ceiling (R² ≈ 0.60 for linear models) is a feature limitation, not algorithmic
  • Target encoding outperforms one-hot for high-cardinality categoricals
  • XGBoost breaks through the ceiling: R² = 0.656 via sequential boosting

Future directions

  • Add photo quality scores via computer vision (currently unmeasured)
  • Incorporate amenities list (WiFi, kitchen, parking) from raw data
  • Model spatial autocorrelation explicitly (e.g. geographically weighted regression)
  • Temporal features: day-of-week, seasonality, proximity to events
Predicting Airbnb prices in NYC is fundamentally a feature quality problem, not a modelling problem.
